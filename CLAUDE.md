# 성요셉목수학교 웹사이트 — CLAUDE.md

## 프로젝트 개요
직업훈련기관(목공·인테리어) 웹사이트 리뉴얼.
- **공개 사이트**: 예비 수강생(주로 40~50대, 모바일·전화 선호)이 과정을 탐색하고 신청·문의
- **어드민(`/admin`)**: 비개발자 운영자가 과정·신청자·문의·게시판을 직접 관리
- 핵심 목표: "이 과정 들으면 진짜 취업되나?"라는 불안을 **보장이 아니라 근거(커리큘럼·자격증·현장·수료후연계)로 해소**

## 기술 스택
- Next.js (App Router) + TypeScript + Tailwind
- Supabase (PostgreSQL + Auth — **관리자만**)
- AWS S3 (+ CloudFront) — 이미지 저장·전송
- Server Actions + zod — 폼 처리·검증
- Resend 등 — 이메일 알림
- 배포: Vercel(Next) + Supabase

## 디렉터리 구조 (제안)
```
app/(public)/          # 공개 페이지 (홈/과정/학원소개/국비지원/훈련사진/상담문의/수강신청)
app/admin/             # 보호 라우트 (운영자 어드민)
lib/supabase/          # Supabase 클라이언트
lib/s3/                # presigned URL 업로드
lib/validations/       # zod 스키마
components/            # 재사용 UI (테이블·폼·카드)
docs/                  # 기획·설계 문서 (source of truth)
```

## ⚠️ 핵심 도메인 규칙 (반드시 준수 — 위반 금지)
계획·구현·리뷰 단계에서 아래를 매번 점검할 것. 이건 의도된 결정이며 "개선"하지 말 것.

1. **회원제 없음** — 일반 사용자 로그인·회원가입·마이페이지를 만들지 말 것. 인증은 `/admin` 관리자만.
2. **취업률 미표시** — 취업률·재취업률 수치를 어디에도 표시 금지(미집계). 신뢰는 후기·자격증·현장으로.
3. **개강일·잔여석 미표시** — 사용자 화면 노출 금지. "신청 시 안내"로 처리. (운영자 내부 데이터로만 보유)
4. **모집상태 수동** — `recruit_status`(모집중/마감)는 운영자 수동 토글. 라이브 자동 판정 만들지 말 것.
5. **신청 유형 분기 (`course.funding_type`)**
   - **A 경기도무료**: 온라인 폼 없음 → 이메일/방문 접수 안내만(신청서 다운로드)
   - **B 국비지원**: 가등록 문의 → 고용24 링크 안내
   - **C 자부담 기능사**: 등록 문의 → 큐넷 링크 안내
6. **외부 시스템(고용24·큐넷)은 링크만** — 연동·결제·완료처리를 흉내 내지 말 것.
7. **인터랙션 패턴** — 수강신청 = 별도 페이지(`/apply`), 상담문의 = 바텀시트.
8. **마감 과정** — 대기신청(waitlist)으로 동작.

## 데이터 모델 (Supabase 테이블)
- 콘텐츠: `course`(+curriculum_item), `exam_schedule`, `post`, `history`, `popup`, `schedule`
- 제출: `application`(가등록/수강신청), `inquiry`(상담), `waitlist`(대기신청)
- 상세 필드·관계는 `docs/데이터정의서_v2_성요셉목수학교.md` 참조. 시드는 `docs/시드데이터_성요셉목수학교.md`.

## 보안 — Supabase RLS (필수, 누락 시 개인정보 노출)
- **공개(anon)**: 게시 콘텐츠 `SELECT` + `application`/`inquiry`/`waitlist` `INSERT`만
- **관리자(authenticated)**: 전체 권한
- 개인정보(이름·연락처) 든 제출 테이블은 **공개 SELECT 절대 금지**

## 코딩 컨벤션
- TypeScript strict. 서버 컴포넌트 기본, 클라이언트 컴포넌트는 필요 시.
- 폼: Server Action + zod 검증. 성공/실패 토스트.
- 이미지: `next/image` + S3/CloudFront URL. 업로드는 presigned URL.
- 모든 데이터 화면은 상태 4종 처리: 기본 / 로딩(스켈레톤) / 빈(대체문구+상담CTA) / 에러(재시도, 전화CTA 유지).
- UI 카피는 **한국어**. 톤: 정중하고 과장 없이.

## 커밋 규칙
- **하나의 기능이 끝날 때마다 커밋한다.** 기능 단위로 작은 커밋을 유지.
- 커밋 메시지는 **Conventional Commits 형식 + 한글 설명**으로 작성한다.
- 형식: `type: 한글 설명`
- type 종류:
  - `feat:` 새 기능
  - `fix:` 버그 수정
  - `chore:` 빌드·설정·잡무
  - `docs:` 문서
  - `refactor:` 리팩터링(동작 변화 없음)
  - `test:` 테스트 추가·수정
  - `style:` 포맷·스타일(동작 변화 없음)
- 예시:
  - `feat: 수강신청 페이지 유형별(A·B·C) 분기 구현`
  - `fix: 문의 폼 연락처 검증 오류 수정`
  - `chore: Supabase RLS 정책 추가`
  - `docs: 화면정의서 대기신청 항목 보강`

## 코딩 행동 가이드라인 (LLM 코딩 실수 줄이기)
> 프로젝트 지침과 함께 적용한다. 트레이드오프: 속도보다 신중함에 무게를 둔다. 사소한 작업은 판단에 맡긴다.

### 1. 코딩 전에 생각하기
추측하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.
구현 전에:
- 가정을 명시적으로 말한다. 불확실하면 묻는다.
- 해석이 여러 개면 모두 제시한다 — 혼자 임의로 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 필요하면 반대 의견을 낸다.
- 불분명하면 멈춘다. 무엇이 헷갈리는지 짚고, 묻는다.

### 2. 단순함 우선
문제를 푸는 최소한의 코드. 추측성 코드는 금지.
- 요청하지 않은 기능 추가 금지.
- 일회성 코드에 추상화 금지.
- 요청하지 않은 "유연성"·"설정 가능성" 금지.
- 일어날 수 없는 시나리오에 대한 에러 처리 금지.
- 200줄 짠 게 50줄로 가능하면 다시 쓴다.
- 자문: "시니어 엔지니어가 이걸 과하게 복잡하다고 할까?" 그렇다면 단순화.

### 3. 외과적 변경
꼭 필요한 곳만 건드린다. 자기가 만든 흔적만 치운다.
기존 코드를 수정할 때:
- 인접 코드·주석·포맷을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 다르게 하고 싶어도 기존 스타일을 따른다.
- 관련 없는 죽은 코드를 발견하면 삭제하지 말고 언급만 한다.
변경으로 고아(orphan)가 생기면:
- 내 변경으로 안 쓰게 된 import·변수·함수는 제거한다.
- 기존에 있던 죽은 코드는 요청 없으면 제거하지 않는다.
- 판단 기준: 바뀐 모든 줄이 사용자 요청과 직접 연결돼야 한다.

### 4. 목표 기반 실행
성공 기준을 정의하고, 검증될 때까지 반복한다.
작업을 검증 가능한 목표로 바꾼다:
- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고, 통과시킨다"
- "X 리팩터링" → "리팩터링 전후로 테스트가 통과하는지 확인한다"
여러 단계 작업은 간단한 계획을 제시한다:
```
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
3. [단계] → 검증: [확인]
```
강한 성공 기준은 독립적으로 반복하게 해준다. 약한 기준("동작하게 해줘")은 계속 되묻게 만든다.

> 이 가이드라인이 잘 작동한다는 신호: diff에 불필요한 변경이 줄고, 과복잡으로 인한 재작성이 줄고, 실수 후가 아니라 구현 전에 질문이 나온다.
