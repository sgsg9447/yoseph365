## 1. 한 줄 요약

> 이 서비스에는 성격이 정반대인 두 사용자가 있다. **전환시켜야 할 외부 사용자(예비 수강생)**와 **운영 부담을 줄여야 할 내부 사용자(학원 운영자)**. 예비 수강생의 진짜 질문은 "비용·일정"이 아니라 **"이 과정을 들으면 정말 취업할 수 있나?"**이며, 운영자의 진짜 문제는 "수정이 번거롭다"가 아니라 **"웹사이트가 실제 업무와 끊겨 있어 무의미했다"**는 것이다.

---

## 2. 사용자 구조 — 페르소나를 몇으로 볼 것인가

목적(재취업/창업·부업/내 집 수리/자격증)으로 페르소나를 쪼개면 디자인이 분산되고, 이들이 막히는 지점(Chapter 1의 14개 문제)은 동일하다. 따라서:

> **주 사용자(Primary) 1명에 최적화하되 그 외를 배제하지 않는다. 목적은 "진입 동기 차이"로 두고 페르소나로 분리하지 않는다.**

| 구분 | 누구 | 처리 방식 |
| --- | --- | --- |
| Primary | 40~50대 중심, 재취업·이직·새 일자리 구직자 | 디자인·동선 최적화의 기준 |
| Secondary | 창업·부업 준비, 자격증 취득, 청년 구직, 내 집 수리 목적 | 목적별 진입점으로 분기, 별도 페르소나로 분리하지 않음 |
| Internal | 학원 운영자(비개발자) | 운영 효율 설계의 대상 |

## 3. 사용자 ① 예비 수강생 (Primary)

> 50대 초반, 이전 직무에서 벗어나 손기술 기반의 새 일자리를 찾는 중. 국민내일배움카드로 비용 부담을 줄이려 한다.

| 항목 | 내용 |
| --- | --- |
| 누구 | 40~50대 중심의 재취업·이직·구직자 (연령은 좁히기 위함이 아니라 주력 사용자 설명) |
| 동기 | 안정적인 기술 직무로의 전환, 국비지원으로 비용 최소화 |
| 디지털 맥락 | 스마트폰 탐색 비중이 높고, 전화 상담을 신뢰·선호하는 경향. 복잡한 가입/로그인보다 빠른 문의를 선호 |
| **핵심 니즈** | **"이 과정을 들으면 내가 진짜 목공을 해서 취업할 수 있나?"** — 실효성·취업 가능성에 대한 확신 |
| 그다음 니즈 | 자비부담금(비용) · 모집/개강 시기 · 신청 방법 |
| 막히는 지점 | 과정·비용 정보가 이미지(문제 4·5·6) / 모집 여부 불명확(7) / 신뢰가 형성되지 않음(12·14) / 신청 후 응답 불투명(9·10) |

## 4. 핵심 설계 과제 — 취업률 없이 확신을 만든다

예비 수강생의 1순위 니즈는 "취업 가능성"이지만, **학원은 취업률을 집계하지 않아 화면에 제시할 수 없다.** 가장 강력한 증거가 없는 상황이다.

기획 원칙은 분명하다.

> **취업을 "보장"하지 않는다. 대신 사용자가 스스로 "할 수 있겠다"고 판단할 "근거"를 충분히 제공한다. 없는 수치를 만들지 않고, 보유 자산으로 확신을 구성한다.**

학원은 신뢰 자산을 이미 갖고 있으나, 작게·이미지로 묻혀 있다(문제 5·8). 이를 재배치한다.

| 신뢰 신호 (보유) | 답하는 사용자 질문 | 현재 상태 → 방향 |
| --- | --- | --- |
| 회차별 커리큘럼 | "막연하지 않고 진짜 배우나?" | 이미지(문제 6) → 데이터화 |
| 자격증 경로(건축목공·도장 기능사) | "검증 가능한 결과물이 있나?" | 과정에 묻힘 → 전면화 |
| 실제 현장 증거(훈련사진·수강일지) | "실제로 이렇게 배우나?" | 작게 노출(문제 8) → 크게·맥락 있게 |
| 수강생 인터뷰·뉴스 영상 | "나 같은 사람도 했나?" | 맥락 보강 |
| 과정 → 직무 연결 | "배우면 무슨 일을 하나?" | 거의 없음 → 신설 |
| 기관 인증(우수훈련기관·이수자평가 A등급) | "믿을 기관인가?" | 보유 (취업 보장으로 과장하지 않음) |

> 사용자의 1순위 니즈(취업 가능성)와 보유 자산(취업률 없음)의 갭을, **보장이 아니라 근거 제공으로 메운다**는 것이 이 프로젝트의 핵심 설계 판단이다.

## 5. 사용자 ② 학원 운영자 (Internal)

> 비개발자. 새 과정·일정·모집상태를 직접 수정하고, 신청자와 문의를 따로 챙겨야 한다.

| 항목 | 내용 |
| --- | --- |
| 누구 | 소규모 운영자(비개발자), 실제 이해관계자 인터뷰 대상 |
| 핵심 니즈 | CMS로 직접 수정 · 신청자 상태를 한눈에 · **웹사이트가 실제 업무에서 편리해야 함** |
| 결정적 맥락 | 수강신청서를 **수기(종이)로 받아 사이트가 사실상 무의미**했음. 사이트가 장식이 아니라 **업무 도구**가 되어야 함 |
| 인터뷰 인용 | *"과정 하나 바꾸려고 이미지를 다시 만든다."* |
| 막히는 지점 | 일정표가 이미지라 재제작·재업로드(문제 6) / 신청서는 종이→스캔→수기 보관(11) / 게시판·이메일로 흩어진 관리 |

운영자의 목표는 "콘텐츠 수정 편의"를 넘어 **"웹사이트를 실제 운영 워크플로우에 연결한다"**로 정의한다. 신청 → 종이 → 스캔 → 수기 보관 대신, 신청 → 데이터 → 상태 관리(접수→상담→등록)로 잇는다.

---

## 6. 사용자 정의가 범위 결정을 정당화한다

Primary가 "40~50대 · 1회성 신청/상담 · 전화 선호 · 디지털 장벽에 민감"으로 정의되면, 회원 기능 제외 결정이 근거를 얻는다.

> 주 사용자의 핵심 행동은 반복 로그인이 아니라 1회성 상담/신청이며 디지털 장벽에 민감하다 → **회원 기능은 전환을 돕는 것이 아니라 가로막는 장벽** → 회원가입/로그인/마이페이지를 제외하고 전화 CTA·간편 신청에 집중한다.

"회원 기능을 왜 뺐는가"라는 질문에 직감이 아니라 사용자 정의로 답할 수 있다.
