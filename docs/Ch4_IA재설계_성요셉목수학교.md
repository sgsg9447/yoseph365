## 1. 한 줄 요약

> 평면적 메뉴·이미지·게시판으로 흩어져 있던 정보를, **"콘텐츠를 데이터로"**(Ch3 메타①)라는 원칙 아래 재구조화했다. 특히 **과정(Course)을 데이터로 만들고 그 안에 "지원 유형"을 속성으로 넣어, 과정마다 다르던 신청 흐름이 데이터에서 자동으로 갈라지게** 했다.

## 2. AS-IS IA — 평면·이미지·게시판 의존

```
메인        배너·영상·사진 위주, 과정 정보 없음            (문제 1·13)
학원소개 ─ 연혁/학원사진/오시는길/경영원칙/미션/비전          (미션·비전 과대 14)
국비지원 ─ 내일배움카드/훈련참여절차/산재훈련 (이미지)         (문제 4)
교육소개[상세커리큘럼] ─ 과정들 + 일정표(이미지)             (문제 5·6)
개강일정안내 ─ 과정카드 → 게시판 → 게시글 → 이메일 접수       (문제 7)
훈련사진 ─ 게시판(작은 썸네일)                             (문제 8)
상담문의 ─ 게시판 + 작성(피드백 없음)                       (문제 9)
수강신청 ─ 폼(상담과 차이 없음)                             (문제 10)
```

## 3. TO-BE 사용자 IA

```
메인  (신뢰 신호 + 모집중 과정 + 전화/신청 CTA)
├─ 학원소개  ┬ 학원소개 (정체성 + 강점 3: 소수정원·현장강사·수료후연계)
│            └ 훈련기관 연혁
├─ 과정 안내 ─ 과정 목록(카드 5개) → 과정 상세(NCS 일정표 + 신청 CTA)
├─ 국비지원  ┬ 국민내일배움카드 안내
│            ├ 훈련참여절차
│            └ 산재노동자 직업훈련
├─ 훈련 사진 ─ 실습 현장 그리드 (+ 수강생 작품)
└─ 상담문의 ─ Q&A(필터: 국비지원/과정/기타) + 전화 강조 + 문의(바텀시트)

[전역 CTA] 수강신청 → 페이지: 모집안내 → 가등록/문의 → 접수완료 (지원 유형별 분기)
```

주요 변화: **개강일정 메뉴 제거**(과정 데이터로 흡수), **학원소개에서 미션·비전 → 강점 3개**(문제 14 해결, 동시에 "수료 후 연계"가 취업 신뢰신호), **국비지원 3개 서브탭**, **모든 콘텐츠가 이미지 → 데이터·텍스트**.

## 4. 핵심 — 과정을 데이터로 (지원 유형을 속성으로)

과정은 더 이상 이미지가 아니라 데이터 레코드다. 그리고 **지원 유형(fundingType)**이 속성으로 들어가 신청 흐름을 결정한다.

| 과정(예시) | 지원 유형 | 신청 흐름 | 웹 이후 절차 |
| --- | --- | --- | --- |
| 인테리어목수 무료 | 경기도 무료 | **A** | 서식 다운 → 이메일/방문 접수 → 면접 → 선발 |
| 주말목공 | 국비지원 | **B** | 전화 등록확인 → 가등록 → 고용24 수강신청·결제 |
| 주말필름 | 국비지원 | **B** | 〃 |
| 집수리 | 국비지원 | **B** | 〃 |
| 도장기능사 | 자부담 | **C** | 전화 등록확인 → 등록 → 큐넷 시험접수 |

> 과정 목록·상세가 흔들려도(과정은 매년 바뀜), **지원 유형 → 신청 흐름의 매핑은 안정적**이다. 그래서 과정별로 따로 만들지 않고 유형으로 분기한다. (Course 속성: name·type·tags·회차/시간·**fundingType**·커리큘럼(NCS)·모집안내·모집상태)

---

## 5. 신청 흐름 분기 — 수강신청 페이지가 유형별로 갈라진다

수강신청 CTA는 하나지만, 클릭한 과정의 `fundingType`에 따라 페이지 내용이 달라진다.

| 단계 | 공통 | A (경기도 무료) | B (국비) | C (자부담 기능사) |
| --- | --- | --- | --- | --- |
| ① 모집안내 | 자격·모집기간·교육일정·진행순서 | 면접→선발 포함 | 고용24 절차 포함 | 큐넷 절차 포함 |
| ② 가등록/문의 | 이름·연락처·관심 과정 폼 | 동일 | 동일 | 동일 |
| ③ 이후 절차 | "1영업일 내 전화" | 서식 다운 + 이메일/방문 접수 | 고용24 링크(외부) | 큐넷 링크(외부) |

핵심 판단:
- **A는 온라인 폼을 만들지 않는다.** 1년 1회성 과정이라 투입 대비 가치가 낮고, 신청서가 자필 필수라 기존 이메일 접수를 유지한다.
- **B·C는 결정적 단계가 외부 정부 시스템(고용24·큐넷)**이라, 웹은 "안내 + 가등록 문의"까지만 한다. "신청 완료"를 흉내내지 않는다(사용자 오도 방지).

## 6. TO-BE 관리자 IA (별도 트리)

```
관리자
├─ 대시보드        신규 신청·문의·모집중 과정
├─ 과정 관리       지원유형·모집상태·비용·기간·커리큘럼
├─ 신청 관리       가등록/문의 접수 → 전화상담 → (유형별 외부/면접 진행) → 등록확인
├─ 문의 관리       Q&A 답변(답변대기→답변완료)
├─ 게시판(훈련사진) 관리
└─ 팝업/배너 관리
```

---

## 7. 공유 데이터 레이어 (두 트리를 잇는 것)

```
Course(지원유형 포함) · Schedule · Application · Inquiry · Post
   ↑ 사용자 화면은 '읽고'              ↑ 관리자 화면은 '쓰고'
   → 같은 데이터를 본다 = 수기 신청서·이미지 수정이 사라지고 사이트가 업무 도구가 되는 지점
```

상세 스키마는 데이터정의서 참조. 핵심은 Course가 `fundingType`을 가져 §5 신청 분기를 데이터로 구동한다는 점.

## 8. 화면 전환 원칙 (모바일 우선)

주 사용자가 40~50대·모바일 중심이라 **UI 무게를 작업 무게에 맞춘다.**

| 작업 | 성격 | 방식 | 이유 |
| --- | --- | --- | --- |
| 수강신청 | 고관여·다단계 | **페이지 전환** | 큰 글씨·키보드 안정·뒤로가기 예측·중단 후 복귀·단계별 측정 |
| 상담문의 | 저관여·1스텝 | **바텀시트** | 맥락 유지하며 빠르게 완료 |
| 전화 연결 | 즉시 액션 | 버튼 | 오버레이 불필요 |

## 9. 핵심 변경점 & 근거

| AS-IS | TO-BE | 근거 |
| --- | --- | --- |
| 개강일정 별도 메뉴 + 게시판 | 과정 데이터로 흡수, 메인 노출 | 메타① / 문제 7 |
| 교육소개 + 일정표 이미지 | 과정상세에 커리큘럼·일정 데이터 집중 | 메타①·② / 문제 5·6 |
| 국비지원 이미지 | 텍스트·지원율 표 + 3 서브탭 | 메타① / 문제 4 |
| 학원소개 미션·비전 과대 | 정체성 + 강점 3개(수료후연계=취업신뢰) | 세부 D / 문제 14 · Ch2 §4 |
| 훈련사진 작은 갤러리 | 라벨 붙은 큰 그리드 | 세부 D / 문제 8 |
| 신청·문의 게시판 전용 | 목적별 분리: 신청=페이지, 문의=바텀시트 + 피드백 | 세부 C·B / 문제 9·10 |
| 과정마다 제각각 신청 | 지원 유형(A·B·C)으로 분기 | 메타① / §5 |
| 회원가입/로그인/마이페이지 | 제거 | Ch2 §6 |
