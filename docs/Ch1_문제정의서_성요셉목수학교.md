| 항목 | 내용 |
| --- | --- |

---

## 1. 한 줄 요약

> 기존 사이트는 정보와 기능을 갖췄지만, **콘텐츠가 데이터가 아닌 이미지·게시글로만 존재**하고 **사용자 행동 흐름과 정보 위계가 설계되지 않아**, 예비 수강생은 신청을 판단·실행하기 어렵고 운영자는 반복 수작업에 묶여 있다.

---

## 2. 분석 방법

실제 운영 중인 사이트를 두 관점에서 점검했다.

| 방법 | 내용 |
| --- | --- |
| AS-IS 휴리스틱 평가 | 예비 수강생 입장에서 메인 → 과정 → 국비지원 → 일정 → 신청까지 직접 사용하며 이탈 지점 기록 |
| 이해관계자 인터뷰 | 실제 학원 운영자를 인터뷰해 콘텐츠 수정 방식·상담 유입·신청자 관리의 반복 업무 파악 |

> 토이 프로젝트가 아니라 **실제 운영 사이트와 실제 이해관계자**를 대상으로 한 분석이라는 점이 이 프로젝트의 출발점이다.

---

## 3. 분석 대상 (AS-IS 개요)

기존 사이트는 빌더 기반으로, 학원 소개·과정 안내·국비지원·개강일정·훈련사진·상담·수강신청 메뉴를 갖추고 있었다. 즉 **"정보가 없는 사이트"가 아니라, 정보와 기능은 있으나 사용자의 신청 판단 흐름과 운영 효율에 맞게 정리되지 않은 사이트**였다.

`[AS-IS 캡처 삽입: 메인 / 학원소개 / 국비지원 / 개강일정 / 신청]`

## 5. 근본 원인 (14개 증상 → 4개 원인)

문제는 많지만, 원인은 넷으로 수렴한다.

| 원인 | 정의 | 수렴되는 증상 |
| --- | --- | --- |
| **A. 콘텐츠가 데이터가 아님** | 과정·일정·소개·비용이 이미지/게시글로만 존재해 비교·수정·상태관리·검색이 불가 | 3·4·5·6·7·8·11·13 |
| **B. 행동·전환 흐름 미설계** | 탐색→판단→신청 동선, CTA 위계, 신청 후 피드백이 없음 | 1·2·9·10 |
| **C. 게시판 전용(轉用)** | 신청·문의·일정에 목적이 다른 게시판을 억지로 사용 | 7·8·9 |
| **D. 정보 우선순위·시각 위계 실패** | 핵심(과정·정체성·신뢰)은 안 보이고 부차(미션/비전·구식 배너)가 큼 → 첫인상에서 기관 신뢰가 형성되지 않음 | 12·13·14 |

## 6. 영향 분석 (왜 문제인가)

각 원인이 누구에게 어떤 손실을 만드는지로 번역한다.

| 관점 | 영향 |
| --- | --- |
| 예비 수강생 | 신청 판단에 필요한 정보(과정·비용·일정·절차)가 분산·이미지화되어 의사결정 중 이탈 가능성이 높음. 구식 인상으로 신뢰가 형성되지 않고, 신청 후 응답이 불투명해 불안이 남음 |
| 운영자 | 일정·비용 수정마다 이미지를 재제작, 신청·문의는 게시판/이메일로 수기 확인, 신청서는 종이 스캔으로 보관 → 반복 업무에 시간이 묶임 |
| 비즈니스 | 신뢰·정보·동선 부재가 곧 **상담·신청 전환 손실**. 운영 비효율은 인건비·기회비용으로 누적 |

> 국비지원 교육기관에서 **신뢰는 신청 전환의 전제 조건**이므로, 첫인상 신뢰 실패(원인 D)는 단순 미관 문제가 아니라 전환 손실로 직결된다.

---

## 7. 핵심 문제 정의

> 예비 수강생은 신청을 결정하기 위해 "내가 들을 수 있는지 · 언제 시작하는지 · 비용은 얼마인지 · 어떻게 신청하는지"를 한 흐름으로 확인하고 싶다. 그러나 기존 사이트는 이 정보가 **이미지·게시글로 흩어져 있고(A) 우선순위가 잘못 잡혀 있으며(D), 신청까지의 동선과 피드백이 설계되지 않아(B) 부적합한 게시판으로 처리(C)**되고 있어, 사용자는 판단·실행에서 이탈하고 운영자는 반복 수작업에 묶인다.

---

## 8. 해결 방향 예고 (→ Chapter 2 연결)

4개 근본 원인을 그대로 뒤집으면 설계 원칙이 된다.

| 근본 원인 | 설계 원칙 (TO-BE) |
| --- | --- |
| A. 콘텐츠가 데이터가 아님 | 과정·일정·신청·문의·게시글을 **구조화된 데이터**로 분리 |
| B. 행동·전환 흐름 미설계 | 탐색→판단→신청 **전환 흐름과 신청 후 피드백**을 설계 |
| C. 게시판 전용 | 신청·문의·일정을 **목적에 맞는 전용 기능**으로 분리 |
| D. 정보 위계 실패 | 핵심(과정·정체성·신뢰)을 **우선 노출하는 정보 위계** 재설계 |

> 다음 챕터에서 사용자 정의 → IA 재설계 → 유저 플로우 → 화면 설계로 이 원칙을 구체화한다.
